## 6. Failures (Detail)

All three failures are pre-existing dev-mode signals, not regressions in
the frontend. Spec fixes were committed in the same session.

### Failure 1 — `e2e/security/csp.spec.ts` › `CSP carries a per-request nonce on style-src`

- **Severity**: S3 (test-side gating)
- **Reproduction**: `pnpm test:e2e:t1` against `pnpm dev`
- **Observed**: dev CSP returns `style-src 'self' 'unsafe-inline'` (matches
  `frontend/middleware.ts` `buildDevCSP`)
- **Expected**: `style-src ... 'nonce-...'` (production CSP)
- **Root cause**: spec did not gate on `NODE_ENV`. Production CSP uses a
  nonce; dev CSP intentionally uses `'unsafe-inline'` for HMR.
- **Fix**: added `test.skip(!isProd, ...)` to the style-src nonce assertion;
  `unsafe-eval` test was already prod-gated.
- **Owner / ticket**: in-session fix (no ticket).

### Failure 2 — `e2e/security/xss-surface.spec.ts` › `page source does not reference dangerouslySetInnerHTML on agent regions`

- **Severity**: S3 (test-side scoping)
- **Reproduction**: `pnpm test:e2e:t1`
- **Observed**: substring `dangerouslySetInnerHTML` appeared in rendered
  HTML (Next.js / next-themes hydration markers, not agent output)
- **Expected**: zero references in agent output regions
- **Root cause**: spec did a global page-content scan instead of scoping
  to `[role='log']` / `article[aria-live='polite']` / `.message-content`.
- **Fix**: rewrote the test to drive a mocked agent response and inspect
  only `innerHTML` of agent output regions for `<script` and inline
  `on*=` event handlers.
- **Owner / ticket**: in-session fix (no ticket).

### Failure 3 — `e2e/smoke.spec.ts` › `Security headers (binary: Are security headers present?)` › `response includes strict CSP and security headers`

- **Severity**: S3 (test-side gating, pre-existing spec)
- **Reproduction**: `pnpm test:e2e:t1`
- **Observed**: dev CSP contains `'unsafe-inline'` and `'unsafe-eval'`
- **Expected**: strict CSP (no unsafe-*)
- **Root cause**: pre-existing spec was always going to fail in dev mode;
  the previous run of the suite was probably against a production build.
- **Fix**: NOT applied in this session — the spec lives in `smoke.spec.ts`
  which is the existing T3 file. Documentation note: run T3 specs against
  `pnpm build && pnpm start`, not `pnpm dev`.
- **Owner / ticket**: follow-up — consider adding `test.skip(!isProd, ...)`
  to the smoke-spec strict-CSP assertion or moving it to the dedicated
  `security/csp.spec.ts`.
