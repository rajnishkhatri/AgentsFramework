## Sealed-Envelope Rule

Per [`AGENT_UI_ADAPTER_PLAN.md` §4.4 / risk R3](../../docs/plan/adapter/AGENT_UI_ADAPTER_PLAN.md) and [`docs/FOUR_LAYER_ARCHITECTURE.md` §Trust Foundation](../../docs/FOUR_LAYER_ARCHITECTURE.md):

> **Any payload carrying a cryptographic signature MUST be passed through the frontend bytewise-unchanged.**

This applies to (non-exhaustive):

- `AgentFacts` — every field except declared "operational metadata" is signed
- `TrustTraceRecord` — every field is signed once emitted
- `PolicyDecision` — the `audit_entry` field is signature-bearing
- Any future model whose schema declares a `signature_hash`, `signed_metadata`, or comparable field

### What "passed through unchanged" means in TypeScript

| Action | Allowed? | Why |
|---|---|---|
| Reading a field of a sealed envelope to render it | ✅ | Read-only access cannot break a signature. |
| Storing the envelope as a single JSON blob in component state | ✅ | The blob is opaque from the UI's perspective. |
| Forwarding the envelope to another adapter route | ✅ | Required for HITL approve/deny round-trips. |
| Mutating any field of the envelope (`facts.status = "active"`) | ❌ | Invalidates the signature; trust gate will reject. |
| Re-serializing via `JSON.stringify(JSON.parse(env))` and sending back | ⚠️ | Key reordering will break a non-canonical signature. **Don't.** |
| Adding new fields ("convenience" properties) | ❌ | Pydantic `extra='forbid'` will reject; but more importantly, you've changed the bytes. |
| Filtering fields out before sending to a downstream adapter | ❌ | Adapter re-validates the envelope; missing required fields fail. |

### How to enforce this in code

1. **Type the envelope as `Readonly<components["schemas"]["AgentFacts"]>`.** TS will refuse field assignment at compile time.
2. **Never spread (`{...envelope}`) a sealed envelope.** Spreading creates a new object whose key order and identity are not guaranteed to round-trip cleanly.
3. **When echoing the envelope back to the adapter, send the *exact original JSON string* the server sent you.** Keep the raw bytes (e.g., the value off `Response.text()`) alongside the parsed object; send the bytes, not `JSON.stringify(parsed)`.
4. **Verify in CI**: a future PR can add a TypeScript-level lint rule (`no-mutate-sealed-envelope`) and a Vitest contract test that round-trips a fixture envelope through the UI's request path and asserts the bytes survived. Both are scoped to FRONTEND_PLAN_V2.

### What happens if you violate the rule

The adapter's [`agent_ui_adapter/translators/sealed_envelope.py`](../../agent_ui_adapter/translators/sealed_envelope.py) re-runs `verify_signature(...)` on every inbound envelope. A mutated envelope produces a `verify_signature(...) -> False` result and the request is rejected at the trust boundary with a `401`/`403` (depending on the route's PEP). No partial state is mutated server-side; the request is treated as if it never arrived.

## Frontend application code

This directory is **only the codegen output + the rule that governs how to use it**. The actual Next.js application (App Router, Tailwind, shadcn/ui, Cognito sign-in, etc.) is planned in [`docs/plan/frontend/FRONTEND_PLAN.md`](../../docs/plan/frontend/FRONTEND_PLAN.md) and lives in a sibling directory once that plan begins execution. Treat anything you put under `frontend/lib/` other than `wire-types.ts` and this README as a violation of the codegen-contract boundary.
